# Provenance Guard

A content-attribution service. A creator submits text; the system estimates whether
it was written by a human or generated by AI, attaches a **confidence-aware
transparency label**, records the decision in a structured audit log, and lets the
creator appeal. Built with Flask + Flask-Limiter + Groq.

Full design spec (architecture diagram, signal rationale, uncertainty model, edge
cases, AI-tool plan) lives in [planning.md](planning.md).

---

## Quick start

```bash
python -m venv .venv
.venv\Scripts\activate            # Windows (PowerShell/CMD)
# source .venv/bin/activate       # macOS/Linux
pip install -r requirements.txt

# .env must contain your Groq key:
#   GROQ_API_KEY=sk-...
python app.py                     # serves on http://127.0.0.1:5000
```

## API

| Method & path | Purpose |
|---|---|
| `POST /submit` | Analyze text → attribution + confidence + transparency label. **Rate limited.** |
| `POST /appeal` | Contest a decision → logs the appeal, sets status `under_review`. |
| `GET /log` | The structured audit log (decisions + appeals). |
| `GET /health` | Liveness check. |

```bash
curl -s -X POST http://127.0.0.1:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text":"<your text>","creator_id":"test-user-1"}'
```
Returns `content_id`, `label_class` (`ai`/`human`/`uncertain`), `confidence`,
`score`, `label_text`, and both signal outputs. Save the `content_id` for appeals.

## Architecture

```
POST /submit → [rate limiter] → validate
   → Signal 1: stylometric (local)   ─┐
   → Signal 2: LLM-as-judge (Groq)   ─┴→ confidence scorer (fuse + disagreement)
   → transparency label builder → audit log (append) → JSON response

POST /appeal → look up original decision → append linked appeal record
             + set status "under_review" → confirmation
```
Full labeled diagrams in [planning.md](planning.md#architecture).

## Detection signals — *why these two*

Two signals from **deliberately different families** so their blind spots don't
overlap. A signal that fools one often won't fool the other, and that is what lets
the combined score be trustworthy.

### Signal 1 — Stylometric analyzer (`detection.py: stylometric_signal`)
Local, deterministic, explainable. Returns `p_ai ∈ [0,1]` from structural features:
- **Burstiness** — variance in sentence length. Humans mix short and long sentences;
  LLM prose is evenly paced.
- **Lexical diversity** — type-token ratio (weak, small weight, length-gated).
- **Transition-phrase density** — connectors LLMs over-use ("moreover",
  "furthermore", "it is important to note", …).
- **Repeated sentence openers**.

*Why this signal:* decoder LLMs optimize for the locally-probable next token, which
produces smooth, low-variance, formulaic prose measurable **without any model** —
cheap, transparent, and reproducible. *Blind spot:* it is purely structural with no
understanding of meaning, so it is unstable on short text and can mis-flag formulaic
human writing (legal, academic, non-native English).

### Signal 2 — LLM-as-judge (`detection.py: llm_signal`, Groq `llama-3.3-70b-versatile`)
Sends the text to a large model with a rubric prompt asking for a calibrated
probability + one-line rationale (`p_ai ∈ [0,1]`). *Why this signal:* it captures the
holistic semantic "assistant voice" — hedging, generic phrasing, coherence patterns —
that simple counters cannot. *Blind spot:* it is **not ground truth**; it is known to
be biased against non-native English writers, can be confidently wrong, is
non-deterministic, and depends on an external API.

### If I were deploying this for real
I would not ship an LLM-as-judge as a load-bearing signal — the bias against
non-native writers is a real fairness liability. I would replace or supplement it
with a token-level **perplexity/log-probability** signal from an open model (a more
objective statistical measure), add watermark detection for models that support it,
calibrate thresholds against a **labeled corpus** rather than by hand, and require
**human review before any user-facing "AI" verdict** rather than only on appeal.

## Confidence scoring & uncertainty — *why this approach*

Combined in `scoring.py`:

```
S          = 0.25·p_style + 0.75·p_llm     # fused AI-likelihood
d          = |p_style − p_llm|             # disagreement between the two signals
distance   = |S − 0.5|·2                   # how decisive S is (0 neutral, 1 extreme)
agreement  = 1 − d
confidence = distance · agreement          # high ONLY if decisive AND agreeing
```
Label bands (disagreement override checked first): `d ≥ 0.40` → **uncertain**; else
`S ≥ 0.70` → **ai**; else `S ≤ 0.30` → **human**; else **uncertain**.

**Why this scoring approach.** Confidence is *not* the AI-likelihood score — it is how
sure the system is of the label it chose. The multiplicative `distance · agreement`
encodes a single principle: **be confident only when the fused score is decisive AND
the two signals agree.** Either being weak drags confidence down. This is what makes
the number reflect *genuine* uncertainty rather than just echoing the raw score. The
disagreement override is the concrete false-positive safeguard: when the LLM flags a
human writer that stylometry reads as human, the signals conflict, the result is
forced to `uncertain`, and the creator sees a hedged label — never a confident
accusation.

**Two real submissions with noticeably different confidence** (captured runs):

| Case | p_style | p_llm | S | **confidence** | label |
|---|---|---|---|---|---|
| Casual human ("cannot believe it rained AGAIN…") | 0.21 | 0.20 | 0.20 | **0.59** | human |
| Formal human on monetary policy (LLM over-flags) | 0.23 | 0.80 | 0.66 | **0.14** | uncertain |

Same system, ~4× difference in confidence. The first: both signals agree the text is
human, so confidence is solid. The second: the LLM alone would wrongly brand it AI
(0.80), but stylometry disagrees (0.23) → `d = 0.57` → the disagreement override
rescues it to `uncertain` at low confidence. This is the meaningful-variation test:
the score is not a constant, and it drops exactly when it should.

**Why the 0.25/0.75 weights** (and how I tested calibration). The M2 spec started at
0.4/0.6. Running the four calibration inputs showed the stylometric signal is
conservative — it rarely emits high AI-magnitudes because clearly-AI text often has
naturally varied sentence lengths — so at 0.4/0.6 clearly-AI text under-scored to
~0.66 and was mislabeled `uncertain`. Reweighting to 0.25/0.75 lets the more-reliable
LLM drive the magnitude while stylometry keeps its real job: the **disagreement veto**.
Verified against all four inputs plus the poet false-positive scenario; every one
lands on the intuitively-correct label. Degraded mode: if Groq is unavailable the
scorer falls back to stylometry alone and caps confidence at 0.60.

## Transparency label — the three variants

The label embeds the confidence **percentage** and a **tier word** (`high` ≥60%,
`moderate` ≥30%, `low` <30%), so the text genuinely changes with the score and never
over-claims. Built by `build_label()` in `labels.py`. Exact text
(`{pct}`/`{tier}` filled at runtime):

**AI variant** (`label_class = ai`):
> ⚠️ Likely AI-generated ({tier} confidence: {pct}%). Our automated analysis
> indicates this text was most likely produced by an AI system. If you created this
> yourself and believe the result is wrong, you can appeal it for human review.

**Human variant** (`label_class = human`):
> ✅ Likely human-written ({tier} confidence: {pct}%). Our automated analysis found
> no strong signs of AI generation in this text.

**Uncertain variant** (`label_class = uncertain`):
> ❓ Attribution uncertain (confidence in a firm call: {pct}%). Our two checks
> disagreed or were inconclusive, so we can't reliably say whether a person or an AI
> wrote this. We've labeled it uncertain rather than guess. If you're the creator,
> you can appeal for human review.

Real rendered examples:
- `⚠️ Likely AI-generated (moderate confidence: 48%). …`
- `✅ Likely human-written (moderate confidence: 59%). …`
- `❓ Attribution uncertain (confidence in a firm call: 14%). …`

The uncertain variant is the deliberate safe default — the failure mode is "admit
uncertainty," never "confidently accuse."

## Appeals workflow

Any creator can contest **any** classification (including `uncertain`).

```bash
curl -s -X POST http://127.0.0.1:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id":"<from /submit>","creator_reasoning":"I wrote this myself..."}'
```
On receipt `POST /appeal`: (1) looks up the original decision by `content_id` (`404`
if unknown); (2) appends an **appeal record** linked to the original
(`original_attribution`, `original_confidence`, `appeal_reasoning`); (3) sets status
**`under_review`**; (4) returns a confirmation with `appeal_id`. No automated
re-classification. A human reviewer opening the queue sees the original label +
confidence, both signal scores, and the creator's reasoning.

## Rate limiting

`POST /submit` is limited via Flask-Limiter (in-memory, keyed by client IP):

```python
@limiter.limit("10 per minute;100 per day")
```
**Reasoning.** A real creator submits a handful of pieces in a sitting, so **10/min**
is generous for legitimate use while stopping a script from flooding the endpoint.
Each submission triggers a **paid, latency-bound Groq call**, so the per-minute cap
protects cost and the upstream API. **100/day** bounds sustained automated abuse that
stays under the per-minute limit while still covering any human's daily workload.
Over the limit returns `429` (JSON). Verified — 12 rapid requests:

```
200 200 200 200 200 200 200 200 200 200 429 429
```

## Known limitations

Honest failure modes, each tied to a **specific property of a signal**:

1. **Minimalist / repetitive human poetry → mislabeled AI-leaning.** The stylometric
   signal treats low sentence-length variance and low lexical diversity as
   AI-uniform. A spare poem with short, repetitive lines and simple vocabulary scores
   high on `ai_burstiness` even though a human wrote it. If the LLM *also* happens to
   read it as generic, both signals agree in the wrong direction and the disagreement
   veto can't save it — it would be labeled AI with real confidence. This is a direct
   consequence of using burstiness as a proxy for authorship.
2. **Non-native / formal English → LLM false positive.** The LLM-as-judge is
   documented to over-flag formal or non-native prose (it reads as "AI-clean"). Our
   disagreement override *mitigates* this (stylometry usually disagrees → uncertain),
   but if the formal text also happens to be structurally uniform, stylometry may
   agree with the LLM and the safeguard fails. The root cause is that neither signal
   measures authorship directly — they measure correlates that non-native writing
   also exhibits.
3. **Short submissions (<~40 words).** Stylometric statistics (variance, TTR) are
   unstable on tiny samples, so Signal 1 becomes noise and the result leans entirely
   on the LLM. Mitigated by a minimum-length floor, not solved.

## Spec reflection

**Where the spec helped.** Writing the architecture diagram and API contract *before*
any code kept the seams consistent — `content_id` was designed into the `/submit`
response and audit entry from day one, so the `/appeal` endpoint dropped in cleanly
in M5 with no rework. More importantly, the §3 false-positive scenario (the
minimalist poet) drove a concrete design decision: the **disagreement override**
exists because the spec forced me to trace what happens when a human's work is
misjudged, before I had written the scorer.

**Where the implementation diverged.** The M2 spec defined confidence with a *split*
formula (`S` for the AI band, `1−S` for human, `1−|S−0.5|·2` for uncertain) and
0.4/0.6 weights. In M4 I replaced it with a single unified formula,
`confidence = |S−0.5|·2 · (1−d)`, and retuned the weights to 0.25/0.75. Why: the split
formula was internally inconsistent (the uncertain row contradicted the mapping text),
and testing showed 0.4/0.6 mislabeled clearly-AI text as uncertain. The unified
formula is both consistent and better expresses the design intent — confidence should
fall when the signals disagree, which the split formula didn't capture. planning.md
was updated to match.

## AI usage

I used an AI coding assistant throughout, guided by planning.md. Specific instances:

1. **Stylometric signal.** I directed it to generate `stylometric_signal(text)` from
   the Signal-1 spec, returning `{p_ai, features}`. It produced a reasonable feature
   set and fixed sub-score weights. I **verified it independently** on a bursty human
   paragraph, an even AI-style paragraph, and a one-liner before wiring it in, and
   confirmed the scores moved in the right direction (human 0.0, AI-ish 0.56).

2. **Confidence fusion — overrode the calibration.** I asked it to implement the
   scoring per spec. When I tested the four calibration inputs, clearly-AI text
   mislabeled as `uncertain`. Rather than accept the plausible-looking output, I had
   it **print both signal scores separately**, found the stylometric signal was
   under-reading AI text, and **overrode the spec weights** (0.4/0.6 → 0.25/0.75) with
   a documented rationale. This is the divergence noted above.

3. **Label text — revised for honesty.** The first label draft hardcoded "Two
   independent checks agreed" into the AI/human variants. Because the `ai` class can
   carry moderate/low confidence (e.g. 0.48), that phrasing over-claimed. I **revised**
   the variants to embed a confidence tier word (high/moderate/low) + percentage so
   the label never asserts more certainty than the score supports.
